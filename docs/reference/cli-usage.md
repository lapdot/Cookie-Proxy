# CookieProxy CLI Usage

## Basic Command

```bash
cookieproxy \
  --cookies ./cookies \
  --url https://example.com/account \
  --output ./page.html
```

If `--output` is omitted, text-like responses are written to stdout. Binary responses such as PDFs require `--output`.

## Flags

- `--cookies <dir>`: directory containing JSON cookie files
- `--url <url>`: target URL to request
- `--output <file>`: write the response body to a file instead of stdout
- `--timeout <ms>`: request timeout in milliseconds, default `30000`
- `--max-redirects <n>`: maximum redirect hops, default `5`
- `--referer <url>`: send an explicit `Referer` header for the request chain
- `--accept-language <value>`: override the browser-like `Accept-Language` header
- `--no-client-hints`: disable the default `sec-ch-ua*` client hint headers
- `--verbose`: print progress and request diagnostics to stderr
- `--debug-cookie-match`: print cookie selection reasoning to stderr

- Default cookies directory:

## Browser-Like Request Behavior

CookieProxy sends a browser-like Chrome/macOS navigation header profile by default for HTML requests. This includes header families such as:

- `User-Agent`
- `Accept`
- `Accept-Language`
- `Upgrade-Insecure-Requests`
- `Cache-Control`
- `Pragma`
- `sec-fetch-*`
- `sec-ch-ua*` unless `--no-client-hints` is used

CookieProxy builds these non-cookie headers separately from the outbound `Cookie` header generated by `tough-cookie`.

`Accept-Encoding` is intentionally not sent right now. This keeps response handling simple and avoids explicit decompression requirements.

## Output Behavior

- Text-like responses, such as HTML and JSON, go to stdout unless `--output` is used
- Binary responses, such as PDFs, require `--output <file>`
- When `--output` is used, CookieProxy writes the response body bytes directly to disk
- CookieProxy reads the final response `Content-Type` header to decide whether stdout is safe
- Diagnostic and debug messages go to stderr
- Logging is powered by `pino` internally, but the CLI prints human-readable plain text

## Saving Artifacts

Use the in-repo `artifacts/` directory for one-off saved retrievals. The directory is ignored by git.

Prefer sortable result folders that start with the local timestamp:

```text
artifacts/{YYYY-MM-DD_HH-mm-ss_TZ}_<site>/<slug>/
```

Example:

```text
artifacts/2026-05-12_18-51-54_PDT_foreignaffairs/can-trump-get-new-nuclear-deal-iran/
```

When saving paired HTML and PDF artifacts, run CookieProxy separately for each URL and capture diagnostics beside the outputs:

```bash
npm run dev -- \
  --url https://www.foreignaffairs.com/united-states/can-trump-get-new-nuclear-deal-iran \
  --output artifacts/2026-05-12_18-51-54_PDT_foreignaffairs/can-trump-get-new-nuclear-deal-iran/article.html \
  --verbose \
  --debug-cookie-match \
  2> artifacts/2026-05-12_18-51-54_PDT_foreignaffairs/can-trump-get-new-nuclear-deal-iran/article.fetch.log

npm run dev -- \
  --url https://www.foreignaffairs.com/system/files/pdf/2026/can-trump-get-a-new-nuclear-deal-with-iran-2026-05-12-10-19.pdf \
  --output artifacts/2026-05-12_18-51-54_PDT_foreignaffairs/can-trump-get-new-nuclear-deal-iran/article.pdf \
  --verbose \
  --debug-cookie-match \
  2> artifacts/2026-05-12_18-51-54_PDT_foreignaffairs/can-trump-get-new-nuclear-deal-iran/pdf.fetch.log
```

For important PDF retrievals, verify the saved file starts with `%PDF-` and check the fetch log for `contentType=application/pdf`.

## Redirect Behavior

- CookieProxy selects one best cookie file for the current URL
- The selected file is loaded into a fresh `tough-cookie` jar for that request
- Browser-like non-cookie headers are rebuilt for each request hop
- Redirects are followed manually
- On each redirect target, CookieProxy reruns cookie-file selection and rebuilds the jar

This means different hops in the same request chain can use different cookie files.

If `--referer` is provided, that explicit `Referer` value is preserved across redirect hops.

## Diagnostics

### `--verbose`

Prints progress and request-level diagnostics such as:

- cookie directory loading
- number of cookie files loaded
- target URL being fetched
- request preparation
- response status and redirect flow

### `--debug-cookie-match`

Prints cookie-selection reasoning such as:

- selected cookie file
- candidate scores
- matched cookie counts
- exact-host and suffix-host matches
- rejected reasons

## Troubleshooting

### No matching cookie file

- Check that the cookie file name resembles the target host
- Check that cookie `domain`, `path`, `secure`, and expiry values make the file usable for the URL

### Redirect loop or max redirects exceeded

- Increase `--max-redirects` if the site legitimately redirects several times
- Inspect stderr with `--verbose` to see which URLs are being followed

### Binary responses

- Use `--output <file>` for PDFs and other binary content
- CookieProxy refuses to print binary responses to stdout
- CookieProxy intentionally does not send `Accept-Encoding`, which keeps response handling simpler and avoids explicit decompression requirements
- Downloading PDFs does not require a compressed transfer encoding

### Malformed cookie JSON

- Confirm the file is valid JSON
- Cookie records with a blank string `name` are skipped automatically
- Missing or non-string `name` values still fail normalization
- Confirm required fields such as `domain` are present and valid strings
- Check expiry values in `expirationDate`, `expires`, or `expiry`

### `127.0.0.1` versus `localhost`

- CookieProxy reselects cookies on redirect targets
- A request that starts on `127.0.0.1` and redirects to `localhost` may switch to a different cookie file
- Provide separate cookie files when both hosts matter
